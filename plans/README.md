# QR Customization - Implementation Plans

Generated by the `improve` skill on 2026-07-11, audited against commit `a24fceea`.
Scope: the QR-code customization feature (`src/components/qr/**`, `src/app/sites/[site]/admin/qr-codes/**`, `src/lib/qr/**`, `src/services/qr-design.service.ts`, `src/app/actions/qr-design.ts`, `qr_designs` table + entitlement trigger).

**Headline**: the feature looks complete but does not deliver end to end. Saving a logo fails every time; saved designs can't be reopened; the "download all tables" PDF encodes the wrong table id, ignores all styling, and never applies per-table assigned designs. ## Recommended sequencing

Ship in two waves:

- **Wave 1 (feature works)**: 001 -> 002 -> 003 -> 004 -> 005. After this, a restaurateur can style a logo, save/reopen it, assign per table, and print correct styled cards with feedback.
- **Wave 2 (finish it)**: 006 -> 007 -> 008 -> 009 -> 010.

Gate every plan through the 5 CI checks (`pnpm typecheck`, `pnpm lint --max-warnings 0`, `pnpm format:check`, `pnpm test`, `pnpm build`) before marking DONE. Visual plans (004, 008) require multi-breakpoint light+dark verification (rule 11).

## Dependency notes

- 002 depends on 001: editing existing designs is meaningless while saving is broken.
- 003 is independent but sequence after 002 to avoid churn on the shared components; it is the highest-value plan (makes "Assign per table" actually change printed output).
- 005 folds into 003's call sites if 003 lands first.
- 006 before 009: template display strings move when dead fields are removed.
- 010 last: tests and dedup assume the export pipeline (003) and config shape (006) are settled.
- 007 is independent; its DB migration must be applied to prod by the maintainer (Supabase MCP `apply_migration`), NOT via `pnpm db:migrate` in CI.

## Product decisions captured (from the audit session)

- **Assign feature**: KEEP and wire it — the printed card per table must reflect its assigned design (plan 003), not remove the feature.
- **Downgrade**: hard-enforce the paywall on the assignment path (plan 007, SEC-03) — no grandfathering.

## Deep-dive addenda (folded into plans)

A follow-up deep trace of the save->load->apply->export cycle confirmed the root cause: the feature was built in two halves that don't talk — the live "customizer" half works, the "persistence" half (service, assignment, resolver) is written/tested/migrated but never re-wired into the read/render UI. All confirmed items map to existing plans (002 load/edit, 003 export+resolve). Two additional items were folded in:

- **Illusory PNG/SVG paywall**: the PDF button exports the same styled DOM for free, so gating PNG/SVG protects nothing -> plan 008 Step 3 (product decision: gate all custom exports, or none; real boundary is save/assign in 007).
- **Dead `resetConfig`**: exported from `useQRDesignConfig` but never consumed -> plan 006 Step 6 (wire a reset button or remove).
- **Print from "Personnaliser" tab prints the whole admin shell**: `QRExportBar`'s Print button calls `window.print()` but `QRPrintSheet` (the `#qr-print-root` isolation CSS) is only mounted in the Download tab -> plan 005 Step 3 (mount the print sheet where Print can fire, or hide the button outside Download).

Cumulative: ~19 findings across 4 audit passes (5 security, 5 surface-functional, 5 deep save/apply, 3 tests, plus print isolation + table-id mismatch), all mapped into plans 001-010. Root cause is singular: persistence/assignment/resolver built and unit-tested but never re-wired into the read/render/export UI, compounded by divergent export paths (StyledQR vs qrcode.react; table_number vs display_name; defaults vs stored config). Storefront confirmed uncoupled (QR encodes only the menu URL; `resolveDesignForTable` has zero consumers anywhere) — so the resolver's intended consumer is the export/print path (plan 003).

## Findings considered and rejected (do not re-audit)

- **XSS in QR export/print**: not a finding. User strings render as escaped React children or into React inline-style objects; `logo.src`/image feed the `qr-code-styling` image loader (not a script sink); no `dangerouslySetInnerHTML` in the QR path. (Input-scheme validation is still tightened in plan 007 as defense-in-depth, not because XSS is reachable.)
- **Cross-tenant IDOR on qr_design_id / tenant_id from client**: not a finding. Actions derive tenant from session (`getAuthenticatedUserWithTenant`), use `getUser` not `getSession`, and the service checks ownership (`assertDesignOwned`) and filters `tenant_id`. SEC-01 is same-tenant privilege drift, not cross-tenant.
- **computeTiling off-by-one / geometry bug**: not a finding. `pdf-tiling.ts` clamps rows/cols and scale correctly; covered by existing tests.
- **buildQRUrl encoding**: correct (uses `URL`/`searchParams`). The bug is the batch path passing the wrong _field_ (display_name), fixed in plan 003 — not an encoding bug.
